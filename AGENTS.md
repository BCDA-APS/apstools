# AGENTS.md

Instructions and conventions for AI coding agents working in this repository.

## Identity

Always identify yourself at the end of every commit message with:

```
Generated by OpenCode (argo/claudesonnet46)
```

Of course, use correct agent and model names.  These are examples.

## Commit Messages

- Format: `<PREFIX> #<issue>: <concise description>`
- The issue number must follow immediately after the prefix, before the description.
- Keep the subject line under 72 characters.
- Add a blank line, then a short body explaining *why* (not just *what*).
- Append the agent identity line after the body.

Common prefixes (following the project's existing style):

| Prefix | When to use |
|--------|-------------|
| `ENH`  | New feature or enhancement |
| `CI`   | Continuous Integration |
| `DOC`  | Documentation only |
| `FIX`  | Bug fix |
| `GIT`  | Git configuration |
| `MNT`  | Maintenance / refactor |
| `TST`  | Tests only |

Example:

```
ENH #1150: add full bluesky run uid to metadata shared with DM

Add ``bluesky_run_uid`` field containing the complete run uid to the
dataset info passed to ``share_bluesky_metadata_with_dm()``, enabling
tiled server filtering by authenticated access using DM.

Generated by OpenCode (argo/claudesonnet46)
```

## Branch Names

- Always start with the issue number, followed by a concise version of the issue title.
- Use hyphens to separate words.
- Format: `<issue>-<concise-title>`

Example: `1150-full-run-uid-to-dm`

## Changelog

- Add an entry to `CHANGES.rst` for every user-visible change.
- Place entries under the appropriate (possibly still-commented-out) upcoming release section.
- Use the correct subsection heading (`Enhancements`, `Fixes`, `Maintenance`, etc.).
- Reference the issue number with the RST role `` (:issue:`<number>`) ``.
- Keep the release date in the upcoming section in sync with the due date on the corresponding GitHub milestone.

Example:

```rst
Enhancements
------------

* Add full bluesky run ``uid`` (as ``bluesky_run_uid``) to metadata shared with APS DM. (:issue:`1150`)
```

## Deprecated Package Tags

This project uses the `deprecated` package (`deprecated.sphinx`) to annotate
public API changes.  Keep these tags consistent with any code changes:

| Decorator | When to use |
|-----------|-------------|
| `@deprecated(version=..., reason=...)` | Public API being removed or replaced |
| `@versionadded(version=...)` | New public API |
| `@versionchanged(version=...)` | Existing public API whose behaviour changes |

These tags apply to **public API code only** — test files do not need them.

## Before Every Commit

- Run `ruff check .` from the repo root (matching what CI runs) and fix any errors before committing.
- The ruff ``target-version`` is set in ``pyproject.toml`` and must match
  ``requires-python``; both are currently ``py310`` / ``>=3.10``.

## Opening Pull Requests

When opening a PR for an issue, copy the following from the issue to the PR:

- **Assignee** — assign the PR to the same user(s) as the issue (not as reviewer)
- **Labels** — apply the same labels
- **Milestone** — set the same milestone
- **Projects** — add to the same projects

## Do Not

- Do not commit or push unless explicitly asked by the user.
- Do not force-push or amend commits that have already been pushed.
- Do not skip pre-commit hooks.
